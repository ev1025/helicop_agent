## 2. #9 — Token-단위 streaming

**문제**: 기존 multi-agent invoke 는 사용자가 10.9 초 동안 빈 화면 → 답변 일괄 표시. ChatGPT 식 점진적 표시 안 됨.

**구현**:
- `QwenLlamaCppChatModel._stream()` — llama.cpp `stream=True` 활용
- `agents/researcher.py`, `agents/procedure.py` 에 `run_streaming()` generator — RAG 검색 후 `final_answer` 도구 wrapping 없이 plain text 토큰 yield
- `routes/chat_v2.py` mode=multi 경로 변경 — Supervisor (non-stream) → Worker.run_streaming → SSE 토큰 전달
- Critic 은 streaming UX 와 충돌 (답변이 다 나와야 평가 가능) → streaming 모드에선 skip

**1차 구현 → 문제 발견 → 수정**:
- 처음엔 `create_chat_completion(stream=True)` 사용 → TTFT 1.4s 로 빨랐으나 (a) chatml-function-calling 핸들러가 13KB RAG 컨텍스트를 일부만 사용 → 답변이 RAG 근거 부실 ("터보프롭/터보팬 엔진" 같은 헬기에 없는 것 언급), (b) chunk 경계의 불완전 UTF-8 byte 를 버려서 한글 글자 누락.
- → `_stream()` 을 `create_completion(stream=True)` + ChatML 프롬프트 직접 구성으로 교체. 전체 RAG 컨텍스트 사용 + UTF-8 누락 해결. 단 13KB prefill 비용 ~5s 가 그대로 노출됨 (8GB GPU 물리 하한 — 기존 1.4s 는 컨텍스트 truncation 의 부작용이었음).

| 지표 | 기존 multi (invoke) | streaming (최종) |
|---|---|---|
| TTFT (첫 토큰) | 10.9s | **6.5s** |
| 총 시간 | 10.9s | 9.7s |
| 답변 품질 | (full 컨텍스트) | full 컨텍스트 + UTF-8 정상 |
| Critic 자동 답변 개선 | ✅ | ❌ (트레이드오프) |

status 메시지 ("🔍 검색 중", "📄 결과 N자") 로 6.5s 대기 체감 완화. 추가 단축은 RAG 컨텍스트 크기 줄이거나 (#3/#4 와 상충), 더 큰 GPU 필요.

## 3. #10 — Langfuse 통합

**왜**: LangGraph 멀티에이전트는 호출 4~6 회 흐름 — 어디서 시간 잡아먹는지 / Critic 이 무엇 보고 점수 매겼는지 / streaming 토큰별 latency 등을 trace 로 봐야 디버깅 가능. AutoRAG 는 RAG 부품 sweep 용이라 이 영역 못 다룸.

**구현**:
- `langfuse_handler.py` — env vars (`LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` / `LANGFUSE_HOST`) 미설정이면 no-op. `.env` 자동 로드. `get_callbacks()` (전역 flat handler), `fresh_callbacks(trace_ctx)`, `start_request_trace()` 제공.
- `agents/supervisor.py` / `researcher.py` / `procedure.py` 의 `route()` / `run_streaming()` 에 `trace_ctx` 파라미터 추가 — 내부에서 `fresh_callbacks(trace_ctx)` 로 callback 생성.
- `routes/chat_v2.py` 가 모든 LLM 호출에 callback 주입, `finally` 에서 root span `.end()` + flush.
- `.env` / `.env.example` + `app/api/main.py` 가 `load_dotenv()` (python-dotenv) — env vars 영구 설정.
- `LANGFUSE_SETUP.md` — cloud (cloud.langfuse.com 무료) / self-host (Docker compose) / `.env` 가이드.
- `app/api/routes/pages.py` — Starlette 신버전 `TemplateResponse(request, name)` API 로 수정 (`/chat` 등 페이지가 구버전 호출로 500 나던 것 해결, 브라우저 테스트 가능하게).
- smoke 테스트 (31_langfuse_smoke.py).

**활성화**: cloud.langfuse.com 에서 키 발급 → `.env` 에 3개 적기 → 서버 재시작. 각 LLM 호출의 input/output/latency 가 Tracing 에 기록됨.

**요청 1개 = trace 1개 — 해결 (Gemini 자문 반영)**:
- 막혔던 이유: ① `start_as_current_observation` 컨텍스트매니저 → FastAPI 가 sync generator 를 threadpool 에서 돌려 매 `.next()` 가 새 contextvars copy → `Failed to detach context` 에러. ② 명시적 `trace_id`+`parent_span_id` 를 fresh `CallbackHandler(trace_context=...)` 로 전달 → langfuse 4.x `CallbackHandler` 가 root run 마다 `propagation_context_manager.__enter__()` 로 자체 OTEL context 를 세워 우리 `trace_context` 를 덮어씀.
- 해결: **`event_stream` 을 `async def` 로 전환** — async generator 는 event loop 에서 돌아 OTEL contextvars 가 yield/await 경계를 유지. `langfuse_handler.request_span()` (= `start_as_current_observation` 래핑) 으로 root span 을 ambient OTEL context 로 세우고, 내부의 sync LLM 호출들(`sv_mod.route`, `worker.run_streaming`, `graph.stream`)을 `anyio.to_thread.run_sync` / `starlette.concurrency.iterate_in_threadpool` 로 감쌈 — 이러면 ambient context 가 worker thread 로 복사돼 `CallbackHandler()` (깡통) 가 자동으로 root span 자식이 됨.
- 에이전트(`supervisor`/`researcher`/`procedure`)에서 `callbacks`/`trace_ctx` 파라미터 제거, 내부에서 `new_callbacks()` (= fresh `CallbackHandler()`) 만 호출. trace 결합은 ambient OTEL context 가 알아서 함.
- 검증: `chat-v2-multi` trace 1개 안에 obs ~10개 nested. researcher/procedure/smalltalk 라우트 모두 OK.

**RAG 파이프라인 trace 노출 추가**: `rag_search.invoke(..., config={"callbacks": new_callbacks()})` 로 도구 호출에 콜백 전달 → `rag_search` TOOL span 생성. `rag_service.py` 의 단계를 `langfuse_handler.maybe_span()`/`end_span()` 으로 계측 — `rag.child_to_parent` (n_child→n_parent, parent_pages), `rag.parent_rerank` (n_after, top_scores; reranker OFF 면 vector 순 상위 K 자르기), `rag.sentence_window` (added, pages). 이제 Tracing 에서 "이 질문: child 60 → parent 57 → rerank 7 → +인접 1 = 8" 까지 펼쳐 보임. `rag_service` 가 `langfuse_handler` 를 import 하지만 env vars 없으면 모두 no-op.

**버그 fix — RAG 결과 JSON bloat → context window 초과**: 광범위 질의("헬리콥터")에서 `Requested tokens (8601) exceed context window of 8192` 에러. 원인: `rag_search` 도구가 결과를 `json.dumps(..., indent=2)` 로 직렬화 — 본문은 `MAX_CONTEXT_LENGTH=8000` 으로 잘 잘려 있는데 각 문서의 metadata dict + JSON 들여쓰기/키 가 붙어 ~15000자로 부풀어 LLM prompt 가 8192 초과. n_ctx 늘리는 건 band-aid 라, `rag_search` 가 LLM 에게 **본문만** (`[문서 N · p.PAGE]` 헤더 + 내용, metadata 없음) 반환하고 누적 `MAX_CONTEXT_LENGTH` 초과분 절단하도록 수정. 결과: "헬리콥터" 15159자 → 8048자, n_ctx 는 8192 유지. (Langfuse `rag_search` span output 도 이 lean text — 여전히 가독성 OK.)

## 4. 의사결정 — autorag 와 우리 평가의 관계

**결론**: 합치지 않음. 단방향 (autorag → 우리) 적용만.

**이유**:
| 도구 | 평가 대상 | 평가 메트릭 |
|---|---|---|
| autorag | RAG 부품 (임베딩/reranker/hybrid) — 단일 LLM 호출 가정 | retrieval F1/recall/NDCG, generation BLEU/ROUGE/sem_score |
| 우리 (#1~#10) | 에이전트 흐름 (Supervisor/Worker/Critic) — 4~6 LLM 호출 | keyword_recall + 답변 본문 + TTFT + 무한루프 횟수 |

같이 평가하면 어느 층위에서 이득 났는지 분리 못 함. autorag 가 결정한 최적 RAG 부품을 우리 `rag_service.py` 에 적용 → 우리 평가 재실행 흐름 권장.
